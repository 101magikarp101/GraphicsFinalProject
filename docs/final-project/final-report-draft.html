Pixelmon-Style Voxel Game with Advanced Shadow Rasterization

1. Goals and Algorithms

This project extends a Minecraft-like voxel renderer into a playable Pixelmon-style graphics demo with original blocky creatures, wild spawning, starter selection, turn-based battles, real-time 3D attacks, progression, and a runtime comparison of voxel ambient occlusion, shadow maps, and stencil shadow volumes.

Ambient occlusion is the baseline. Each visible cube carries per-face vertex occlusion from neighboring voxel occupancy. The fragment shader bilinearly interpolates four values and applies AO = mix(0.3, 1.0, pow(clamp(ao / 3, 0, 1), 0.75)). It is stable and cheap, but it only approximates local contact darkness.

Shadow mapping uses a directional light-space camera. A fragment position p is transformed by LVP * p, compared against a depth texture with slope-scaled bias, and filtered with 3x3 PCF. The shadow-strength slider controls direct-light attenuation with direct = mix(1 - strength, 1, visibility).

Stencil shadow volumes use the z-fail method. The sun/moon is a normalized directional light. For each selected caster cube, faces with dot(n, l) > 0 form the front cap; the same cap is extruded in direction -l; silhouette edges are detected where adjacent face signs differ and extruded into quads. The closed volume is rendered into the stencil buffer with color/depth writes disabled. Back faces increment on depth fail, front faces decrement on depth fail, and pixels with nonzero stencil are darkened by a full-screen multiply pass that preserves the original block hue.

References: NVIDIA GPU Gems 3 Chapter 11, Paul’s Projects OpenGL shadow volume notes, and OpenGL SIGGRAPH 1999 stencil shadow volume notes.

2. Implementation Engineering, Results, and Limitations

The renderer is centered on RenderView in src/client/engine/render/renderer.ts. Normal terrain uses blankCube.vert/frag, shadow mapping uses shadowMap.vert/frag, and stencil volumes use shadowVolume.vert/frag plus shadowOverlay.vert/frag. The diagnostics HUD exposes runtime technique and strength controls, and preferences persist them.

The shadow-volume geometry builder is isolated in src/client/engine/render/shadow-volume.ts, and the caster filter is isolated in src/client/engine/render/shadow-volume-casters.ts. Unit tests verify closure, outward winding, extrusion direction, and that the optimized caster filter keeps vegetation and silhouette columns while dropping broad flat terrain. Benchmark exports include JSON, CSV, Markdown, and SVG. Static comparison artifacts are in docs/final-project/; runtime capture is implemented by scripts/capture-shadow-benchmarks.mjs.

Known limitations: shadow maps currently use terrain cubes as the main caster/receiver path. Shadow volumes cast from visible terrain cubes and darken the rendered scene through a stencil mask, but the far cap is finite rather than truly at infinity because WebGL lacks desktop depth-clamp behavior. Both advanced techniques are hard-shadow methods; soft shadows are only approximated by PCF in the shadow-map path. Gameplay is demo-ready, but balance and expansion to all 18 creature types remain future work.
